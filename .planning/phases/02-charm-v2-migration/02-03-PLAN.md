  <action>
Rewrite wizard.go from scratch using a BubbleTea v2 Model with a step-based state machine. The WizardResult struct and all provider/package constants MUST be preserved exactly (they are the output contract consumed by Python).

**Architecture:**
- Main `WizardModel` struct implementing tea.Model with `step wizardStep` field
- Per-step sub-models: `multiSelectModel`, `singleSelectModel`, `textinput.Model` (from bubbles v2), `confirmModel`
- Step navigation: Enter advances, Esc goes back (or cancels on step 0)
- On final step completion, write JSON to stdout and tea.Quit

**Step implementations:**

1. **stepAgentPackages (MultiSelect):** Custom model with `items []string`, `cursor int`, `selected map[int]bool`. Space toggles selection, Enter confirms. Render: each item with cursor indicator and [x]/[ ] checkbox. Use theme colors.

2. **stepProvider (SingleSelect):** Custom model with `items []string`, `cursor int`. Up/Down moves cursor, Enter selects. Render: cursor indicator on current item.

3. **stepAPIKey (TextInput):** Use `textinput.Model` from `charm.land/bubbles/v2/textinput`. Set EchoMode to EchoPassword for API keys. Enter submits.

4. **stepProfile (SingleSelect):** Same singleSelectModel as step 2 but with profile options. Only shown for Anthropic/Bedrock providers.

5. **stepOptionalKeys (Confirm + TextInput):** Sequence of Y/N prompts for NCBI, Cloud, Smart Standardization. Custom confirm: render "Y/N" prompt, y/n keys toggle, Enter confirms. If confirmed, show textinput for the key. Track sub-step within this step.

**Ollama special case:** If provider is "ollama", step 3 uses `os/exec` to detect running models (existing logic). Step 4 shows model selection instead of profile selection.

**Key behavior to preserve:**
- Same WizardResult JSON output format
- Same provider list and logic
- Cancellation writes `{"cancelled":true}` to stdout and exits 1
- Title rendering with lobster theme colors

Rewrite forms.go to handle protocol form requests WITHOUT huh and WITHOUT tea.Exec suspension.

**New approach:** Instead of suspending the TUI with tea.Exec to run a huh form, use an inline BubbleTea model approach:
- When a form request arrives, set `m.activeForm` to a new `formModel` that handles the form fields
- The formModel renders in the chat area (or footer area) as a sequence of fields
- On submit, formModel produces formResult message and clears itself
- On cancel (Esc), formModel produces formResult with error

**formModel implementation:**
- Fields: `[]formFieldModel` (one per FormField from protocol)
- Current field index, cursor navigation
- Each field type maps to a sub-component:
  - "text"/"" -> textinput.Model (bubbles v2)
  - "password" -> textinput.Model with EchoPassword
  - "select" -> singleSelectModel (reuse from wizard or inline simple version)
  - "confirm" -> simple Y/N
- Enter on last field submits, Tab/Enter advances to next field
- Esc cancels entire form

**Key changes to model.go integration:**
- The existing `runFormSuspended` method is replaced
- In Update(), handle formResult message the same way as before (send response to protocol handler)
- The formExecCmd struct is removed (no more tea.Exec)
