<feature>
  <name>AquadifMonitor Service Class</name>
  <files>lobster/core/aquadif_monitor.py, tests/unit/core/test_aquadif_monitor.py</files>
  <behavior>
    record_tool_invocation:
    - Given tool_metadata_map has "analyze_cells" with categories=["ANALYZE"], provenance=True
      When record_tool_invocation("analyze_cells", "transcriptomics_expert") is called
      Then category_counts["ANALYZE"] == 1 and tool_invocation_counts["analyze_cells"] == 1
    - Given tool has categories=["ANALYZE", "FILTER"] (multi-category)
      When record_tool_invocation is called
      Then both "ANALYZE" and "FILTER" category counts increment
    - Given tool has categories=["CODE_EXEC"]
      When record_tool_invocation is called
      Then code_exec_log has 1 entry with tool_name, timestamp (ISO format), and agent
    - Given tool_name not in tool_metadata_map (unknown tool)
      When record_tool_invocation is called
      Then no crash, no category counts change, no code_exec_log entry
    - Given tool has provenance=True and not yet seen
      When record_tool_invocation is called
      Then provenance_status[tool_name] == "missing" (pre-set, awaiting record_provenance_call)
    - Given record_tool_invocation raises internally
      Then exception is swallowed (fail-open), no re-raise

    record_provenance_call:
    - Given tool was invoked (provenance_status == "missing")
      When record_provenance_call(tool_name, has_real_ir=True) is called
      Then provenance_status[tool_name] == "real_ir"
    - Given tool was invoked
      When record_provenance_call(tool_name, has_real_ir=False) is called (ir=None bridge)
      Then provenance_status[tool_name] == "hollow_ir"
    - Given tool already has "real_ir" status
      When record_provenance_call(tool_name, has_real_ir=False) is called
      Then provenance_status stays "real_ir" (real_ir wins over hollow_ir)

    get_category_distribution:
    - Returns dict copy of category counts (not a reference)
    - After 3 ANALYZE + 2 IMPORT calls -> {"ANALYZE": 3, "IMPORT": 2}

    get_provenance_status:
    - Returns {"real_ir": [...], "hollow_ir": [...], "missing": [...]} grouping
    - Tools correctly sorted into their status groups

    get_code_exec_log:
    - Returns list (snapshot) of CodeExecEntry objects
    - Bounded to maxlen=100 — 101st entry evicts 1st
    - Each entry has tool_name, timestamp (ISO string), agent (string)

    get_session_summary:
    - Returns structured dict with: category_distribution, provenance_status, code_exec_count, code_exec_log (serialized), total_invocations
    - Consistent snapshot (lock-protected)

    Thread safety:
    - Concurrent record_tool_invocation from multiple threads does not corrupt state
    - get_session_summary during concurrent writes returns consistent snapshot

    Edge cases:
    - Empty tool_metadata_map -> all methods work, no crashes
    - record_provenance_call for tool not in map -> no crash (fail-open)
  </behavior>
  <implementation>
    Implement AquadifMonitor as a pure-stdlib class (zero lobster imports) following the exact design from 05-RESEARCH.md Pattern 1. Use threading.Lock for compound mutations, collections.deque(maxlen=100) for CODE_EXEC log, simple dicts for counters. CodeExecEntry as a dataclass. All public methods wrapped in try/except for fail-open behavior.

    Key implementation decisions:
    - Module has ZERO imports from lobster.* (prevents import cycles when callbacks.py imports it)
    - Use string comparison for CODE_EXEC detection ("CODE_EXEC" in categories), not enum import
    - Lock only compound mutations (deque append + counter increment in record_tool_invocation, dict iteration in get_session_summary/get_provenance_status)
    - Simple dict increments (self._category_counts[cat] = self._category_counts.get(cat, 0) + 1) are GIL-safe for single ops, but wrap in lock when part of a compound operation
  </implementation>
</feature>
